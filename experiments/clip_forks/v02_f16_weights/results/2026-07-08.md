# v02 F16 Weights Result

Date: 2026-07-08

## Summary

Implemented a gated CLIP `PRECISION=f16` / `CLIP_PRECISION=f16` weights-only
path for tools and 3D step benches. It preserves f32 activations, gradients,
workgroup tiles, reductions, and accumulators.

Result: do not promote this all-weight f16 path as default.

Reasons:

- f16 assets pack correctly and halve payload size;
- f16 WGSL compiles and runs with `shader-f16`;
- final embedding is very close to f32;
- input-gradient cosine fails the planned gate;
- stable isolated CLIP timestamps were slower in this run;
- integrated 3D timestamp was slower in this run.

## Implemented Surface

- `tools/clip/pack_f16_weights.ts`
- `tools/clip/f16_compare.ts`
- `PRECISION=f16` in `tools/clip/dispatch_profile.ts`
- `PRECISION=f16` in `tools/clip/batch_major_train_bench.ts`
- `CLIP_PRECISION=f16` in `tools/splat3d/step_bench.ts`
- precision-aware CLIP WGSL `weightsDecl()`
- `Float32Array | Uint16Array` runtime weight uploads
- 3D optimizer `clipWeightPrecision` option

Default f32 behavior remains unchanged.


Output:

```text

inputGrad : cos=0.97493807 relLinf=3.933e-1 maxDiff=3.462e-4 norm(f16/f32)=1.601e-2/1.567e-2
```

The planned gate was:

```text
embedding cosine >= 0.9995
input-gradient cosine >= 0.995
```

Embedding passes. Input gradient fails.

## CLIP Dispatch Timing

Commands:

```bash

PRECISION=f16 TIMESTAMP=1 STEM_SPATIAL_BWD=1 FUSE_PW_GELU=1 MODE=train BATCH=3 RUNS=5 WARMUP=3 bun tools/clip/dispatch_profile.ts
```

Results:

| Precision | Total Isolated Median Sum |
| --- | ---: |
| f32 | `67.109 ms` |
| f16 weights | `71.107 ms` |

In this stable short run, all-weight f16 was slower by about `6.0%`.

## Decision

Keep the f16 path as an experiment tool, not a promoted runtime default.

Next f16 direction, if we revisit it:

1. Do not convert every CLIP weight to f16 blindly.
2. Try selective f16 on pointwise-heavy weights only with two weight buffers, or
   target storage/layout where timestamp and future Metal counters show memory
   pressure.
3. Keep f32 for numerically sensitive late/head/SE/attention pieces until a
   per-family comparison proves they are safe.

The larger speed path is still likely N-of-K/grid scheduling plus targeted
pointwise/spatial kernel variants, not naive all-weight f16.
