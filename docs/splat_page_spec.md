# Spec — prompt→splats browser page (Task #7, phase 2)

The end goal: **one page where you type a prompt and 200K Gaussian splats
live-optimize on a canvas to match it.** Phase 1 — the differentiable core —
is DONE and verified end-to-end headless; this phase is the browser wrap only.

## What already works (do NOT rebuild — just wrap it)

`src/splat/optimize.ts` → `SplatOptimizer`, verified by
`bun tools/splat/optimize_test.ts` (cos to "a photo of a cat" rose
0.19→0.31, ranked above dog/diagram). Its API — everything is on ONE shared
`GPUDevice`, GPU-resident:
- `static async create(device, trainPlan, weights: Float32Array, {G?, cap?, bg?, seed?, lrs?, hyper?}) → SplatOptimizer`
- `setPrompt(textEmbed: Float32Array /*512*/)` — call per prompt change
- `step()` — one optimize step, one submit (raster fwd → CLIP fwd+loss+bwd → raster bwd → Adam)
- `renderImage(): Promise<Float32Array>` — current splats as NCHW planar [3][256][256] in [0,1]
- `currentEmbedding(): Promise<Float32Array>` — CLIP embedding of current canvas (for the live cosine readout)
- `cosine(a,b)` helper exported alongside
- `raster.image` is the live GPUBuffer (STORAGE|COPY_SRC), NCHW planar
  `img[c*256*256 + y*256 + x]` — bind it read-only in a fragment shader to blit.

`TrainPlan` type + `VisionTrainer` are in `src/clip/vision.ts`. The plan +
weights files: `models/mobileclip_s0/plan_train.json` (90 KB) and
`weights_train.bin` (82 MB), regenerated by
`uv run --with onnx --with numpy python tools/clip/compile_plan.py --train`.

## Build these

### 1. Second parcel entry
- `package.json`: change `"source": "src/index.html"` to
  `"source": ["src/index.html", "src/splat.html"]` (parcel multi-entry).
  Leave the existing particle-art page untouched and working.
- `src/splat.html` → a module script `src/splat_page.ts` (plain TS + DOM, no
  React needed — keep it lean). Layout: a `<canvas width=256 height=256>`
  (CSS-scaled up, e.g. 512²), a text `<input>` + "Optimize" button below it,
  a live readout line (step count, cos-to-prompt), and a Reset button.

### 2. WebGPU + optimizer boot (`src/splat_page.ts`)
- Own device: `adapter = await navigator.gpu.requestAdapter(); device = await adapter.requestDevice()`.
  NO tfjs. If `!navigator.gpu`, show a "needs WebGPU" notice (match the
  existing app's behaviour).
- Configure the canvas context on THAT device
  (`canvas.getContext("webgpu").configure({device, format: navigator.gpu.getPreferredCanvasFormat(), ...})`).
- Fetch `plan_train.json` (JSON) and `weights_train.bin` (arrayBuffer →
  Float32Array). See §4 for serving these.
- `opt = await SplatOptimizer.create(device, plan, weights, { G: 200_000 })`.

### 3. Text encoding (once per prompt, `@huggingface/transformers` — it wraps
ort-web, so this IS the "ORT-web once per prompt" from the plan)
- Use `AutoTokenizer` + the CLIP text model to produce a 512-d embedding, the
  SAME model as `tools/clip/text_onnx.mjs` (read it — it documents the exact
  tokenizer + model behaviour you must match): `Xenova/mobileclip_s0`,
  77-token context, pad with token id 0 after EOT, output is float32 [1,512],
  NOT L2-normalized. Simplest path: `pipeline("zero-shot-image-classification"
  , "Xenova/mobileclip_s0")` is overkill — instead load the tokenizer +
  `CLIPTextModelWithProjection` (or run `text_model.onnx` via the tokenizer's
  `input_ids`) and read the projected embedding. Let transformers.js fetch the
  text model + tokenizer from the HF hub at runtime (cached by the browser) —
  ONLY the CLIP vision train-weights need local serving (§4).
- On "Optimize" (or Enter): tokenize+encode → `Float32Array(512)` →
  `opt.setPrompt(emb)`, reset the step counter display, ensure the rAF loop is
  running. Show a "encoding…" state while the model loads (first use is slow).

### 4. Serving the vision weights for dev
`models/` is gitignored and outside `src/`, so parcel won't serve it and you
must NOT bundle 82 MB through parcel. Pick the simplest robust mechanism and
DOCUMENT the run command in the page's header comment + README:
- e.g. a tiny static server (copy the pattern from `tools/clip/ort_web_bench.mjs`,
  which already serves the repo root + the built page over one node http
  server), or copy the two files into a parcel-served `static/`-style dir, or
  symlink. Whatever you choose, `fetch()` of the plan + weights must work in
  the running page. The puppeteer gate (§6) must exercise the real fetch.

### 5. Display blit + loop
- A render pipeline: fullscreen triangle (3 verts, no vertex buffer), fragment
  shader binds `opt.raster.image` as `var<storage, read> img : array<f32>` and
  writes `vec4(img[0*HW+i], img[1*HW+i], img[2*HW+i], 1)` for pixel i=y*256+x
  (flip Y if it looks upside down). Build it once on the page's device.
- `requestAnimationFrame` loop: while running, call `opt.step()` a few times
  per frame (tune so the page stays responsive — e.g. 1–2 steps/frame), then
  encode the blit pass to the canvas texture and submit. Every ~15 steps,
  `await opt.currentEmbedding()` → update the cosine readout (don't block every
  frame on the readback).
- Reset button: rebuild the optimizer (new random seed) and clear the prompt.

## Acceptance gate — `tools/splat/page_smoke.mjs` (puppeteer, REAL GPU)
Adapt `tools/qa_browser.mjs` (headless Chrome, `--use-angle=metal`,
`--enable-webgpu-developer-features`). It MUST, against the actually-running
page (with weights served per §4):
1. load the page, assert no page errors and `navigator.gpu` present;
2. type a prompt ("a photo of a cat"), click Optimize;
3. wait for the text model to load + ~150 optimize steps to run;
4. read the cosine readout and assert it ROSE from its initial value;
5. screenshot the canvas and assert it is NON-BLANK (pixel variance above a
   floor — a blank/constant canvas fails);
6. print SCREENSHOT path + the cos trajectory.
Do not report success until this smoke passes on the real GPU. Include the run
commands (serve + parcel build/serve + smoke) in your final message.

  `main.ts`, `src/render/`), the CLIP kernels, or the splat raster kernels.
  You are wiring verified pieces + DOM + a blit shader only.
- No new optimization math, no densification, no fp16 — just the wrap.
- Keep `src/splat/optimize.ts`'s API; if you truly need a new getter, add it
  minimally and keep `bun tools/splat/optimize_test.ts` green.
- Update `tools/clip/README.md` (or a new `docs` note) with how to run the page.
