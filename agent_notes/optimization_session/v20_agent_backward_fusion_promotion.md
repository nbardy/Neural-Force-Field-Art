# v20 Agent Note - Backward Fusion Promotion

Date: 2026-07-08

## Scope

Question: inspect the current v11 backward local fusion code and docs, then
answer whether `FUSE_GELU_BWD_PW` and `FUSE_RESIDUAL_BWD_PW` are safe to promote
by default, and under which exact runtime conditions.

This note is documentation-only. No source code was edited.

## Verdict

Do not promote these as unconditional global defaults.

They are safe to promote as a paired hidden default only for the measured
grid/direct-raster/depthwise batch path:

- model/runtime: current `models/mobileclip_s0/plan_train.json` train plan,
  generated by `tools/clip/compile_plan.py --train`, with frozen CLIP weights and
  the current activation/grad slot convention;
- precision: f32 CLIP weights (`weights_train.bin`), not the f16 weight path;
- CLIP runtime: `BatchMajorVisionTrainer` or `VisionTrainer` using current
  `planBwdDispatches()` dispatch specs;
- UI/optimizer schedule: `clipLayout === "grid9_close2"`;
- batch: `clipBatchSize === 3`;
- views: the grid path's 9-view/contact-sheet schedule, as benchmarked with
  `VIEWS=9`;
- raster path: `gridDirectRaster === true`;
- backward spatial option: `stemSpatialBwd === true` and
  `spatialBwdVariant === "depthwise4"`;
- forward pointwise/GELU option: `fusePointwiseGeluForward === true`;
- promotion shape: enable `fuseGeluBwdIntoPw` and `fuseResidualBwdIntoPw`
  together, keeping an env/config escape hatch for negative-control benches.

That is narrower than "turn both booleans on everywhere." The current app default
is still per-view CLIP with `gridDirectRaster=false`, and
`Splat3DOptimizer.create()` still defaults both backward fusions to false. The
evidence supports a conditional grid80+depthwise default after that path is
chosen, not a blanket app-wide default.

## Performance Evidence

v11's measured target stack was:

```bash
TRIALS=7 \
CONFIGS=grid80dw4=9:3:grid9:directgrid:dw4,grid80dw4both=9:3:grid9:directgrid:dw4:gelubwd:resbwd \
RUNS=7 WARMUP=5 \
bun tools/splat3d/step_matrix.ts
```

Recorded result:

| Variant | Normal Median | Profile Median | CLIP Median | Raster Median |
| --- | ---: | ---: | ---: | ---: |
| `grid80 + depthwise4` | `56.20 ms` | `58.33 ms` | `36.41 ms` | `19.51 ms` |
| `grid80 + depthwise4 + both fusions` | `54.63 ms` | `56.81 ms` | `34.91 ms` | `19.29 ms` |

That is a real but small win:

- normal step: about `1.57 ms`, roughly `2.8%`;
- CLIP median: about `1.50 ms`, roughly `4.1%`;
- profile total: about `1.52 ms`, roughly `2.6%`.

The isolated timestamp profile in v11 reported dispatch count `257 -> 211` and
timestamp sum `86.704 ms -> 54.591 ms`, but the integrated step medians are the
numbers to trust for promotion.

## Exact Promotion Recommendation

If source defaults are changed later, prefer conditional defaulting at the
`Splat3DOptimizer` option layer rather than changing the low-level default inside
`planBwdDispatches()`.

The default should behave like:

```ts
const promoteBackwardLocalFusion =
  clipLayout === "grid9_close2" &&
  clipBatchSize === 3 &&
  (cfg.gridDirectRaster ?? false) === true &&
  (cfg.spatialBwdVariant ?? "generic") === "depthwise4" &&
  (cfg.stemSpatialBwd ?? true) === true &&
  (cfg.fusePointwiseGeluForward ?? true) === true &&
  (cfg.clipWeightPrecision ?? "f32") === "f32";

fuseGeluBwdIntoPw = cfg.fuseGeluBwdIntoPw ?? promoteBackwardLocalFusion;
fuseResidualBwdIntoPw = cfg.fuseResidualBwdIntoPw ?? promoteBackwardLocalFusion;
```

If `depthwise4` itself becomes a grid/batch default later, the condition can
treat the effective default as `depthwise4`. With today's code, `spatialBwdVariant`
defaults to `undefined`, so promoting the backward fusions under a strict
`depthwise4` condition would not affect the current app boot path unless the page
or optimizer also opts into depthwise4.

Keep tool/env overrides:

- `FUSE_GELU_BWD_PW=0/1`
- `FUSE_RESIDUAL_BWD_PW=0/1`
- `SPATIAL_BWD_VARIANT=generic/depthwise4`
- `FUSE_PW_GELU=0/1`

The pair should be promoted together. The older GELU-only result was not enough
for a broad default, and the paired result is the measured v11 configuration.

## Risks

1. Current correctness docs are partially stale.

   `tools/clip/bwd_test.ts` does not wire the fusion env flags into
   `VisionTrainer.create()`. The v11 README and perf notes say the env-flagged
   `bwd_test` passed; as written, that command validates the unfused trainer path.
   Before a source promotion lands, update the test harness or add a differential
   fused-vs-unfused gate.

2. The performance win is small and stack-specific.

   The newer paired-v11 result is positive on grid80+depthwise, but the older
   GELU-only integrated result was flat enough to keep gated. Do not use the
   isolated timestamp drop as the promotion argument; use the integrated medians.

3. Do not infer safety for all schedules.

   Correctness of the local rewrite is source-predicate guarded, but performance
   is not proven for per-view CLIP, `clipBatchSize=1`, non-direct grid raster,
   generic spatial backward, cached-gradient cadence, f16 weights, or future
   pointwise tile forks.

4. Current plan assumptions matter.

   The source predicates prove local shape/slot adjacency, not a full arbitrary
   DAG use-count proof. They are safe for the current MobileCLIP-S0 train plan,
   where the adjacent fusable slots have no alias issues and no extra readers.
   A future compiler/model should rerun the static pair and alias checks.

5. Residual fusion depends on pointwise tile coverage.

   `pw.cout >= pw.cin` is required because the residual copy is emitted from the
   pointwise output-channel loop. If a future `pw_bwd` tile variant changes the
   loop shape or write coverage, the residual `extraStore` must be re-audited.

6. Batch-major shader rewriting is string-based.

   `batchSpec()` rewrites storage buffer array accesses by variable name and
   offsets them per batch lane. It works for the current fused specs and passed
   the differential check, but future WGSL formatting or helper changes could
   break the regex assumptions.

7. Keep f16 separate.

   The fused code can call `weightsDecl(..., "f16")`, but prior notes reject the
   all-weight f16 path on gradient-quality/speed grounds. Do not bundle backward
   fusion promotion with f16 promotion.

## Bottom Line

Promote both backward local fusions only as a conditional hidden default for the
grid9 close-up/direct-raster/depthwise4/f32/batch-3 path, after preserving an
override for benchmark comparisons. They are not ready as unconditional defaults
for every CLIP training path.
