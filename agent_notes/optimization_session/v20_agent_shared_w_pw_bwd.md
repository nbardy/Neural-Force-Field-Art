## Current Baseline

`pw_bwd` is generated by `pwBwd()` in `src/clip/vision_bwd_wgsl.ts`. It computes
the frozen-weight input gradient:

```text
dX[co, p] = sum_ci dY[ci, p] * W_T[ci, co]
```

The generated shader uses the same tiled pointwise body as forward pointwise:

```text
workgroup_size: 8 x 8 = 64 threads
tile:           8 pixel-quads x 32 output channels
xS:             256 vec4f
wS:             256 vec4f
workgroup mem:  8192 bytes
workgroups:     [P4 / 8, cout / 32, 1]
```

In batch-major train mode, `batchSpec()` in `src/clip/vision_batch_wgsl.ts`
wraps this into:

```text
workgroups: [P4 / 8, cout / 32, batch]
```

Every batch lane is an independent workgroup-z slice, so every lane stages the
same `wS` tile from the same frozen weights. That is the waste this fork
targets.

## Shared-W Backward Design

Add a batch-aware `pw_bwd` emitter beside the existing forward shared-W emitter
in `src/clip/vision_batch_pointwise.ts`.

Workgroup layout:

```text
@workgroup_size(8, 8, B)
xS: 256 * B vec4f
wS: 256 vec4f
workgroups: [P4 / 8, cout / 32, 1]
```

For `B=3`, workgroup memory is:

```text
xS = 256 * 3 * 16 = 12288 bytes
wS = 256 * 16     = 4096 bytes
total             = 16384 bytes
```

Reuse pattern inside each `ci0 += 32` tile:

```wgsl
let lane = lid.z;
let li = lid.y * 8u + lid.x;       // do not use local_invocation_index
let xTile = lane * 256u;
let srcBase = lane * srcStride;
let dstBase = lane * dstStride;

for (var t = li; t < 256u; t = t + 64u) {
  let ci = t >> 3u;
  let px = t & 7u;
  xS[xTile + t] = src[srcBase + (ci0 + ci) * P4 + p4base + px];
  if (lane == 0u) {
    wS[t] = W4((wOffT + (ci0 + ci) * cout + cobase + px * 4u) / 4u);
  }
}
workgroupBarrier();
```

Then every lane performs the same f32 `fma` sequence as baseline:

```wgsl
let xv = xS[xTile + ci * 8u + lid.x];
let wv = wS[ci * 8u + lid.y];
acc0 = fma(vec4f(wv.x), xv, acc0);
...
```

Store with the existing `accumulate` semantics, but add the per-lane destination
base:

```wgsl
let i0 = dstBase + co * P4 + p4;
dst[i0] = accumulate ? dst[i0] + acc0 : acc0;
```

This is exact math relative to the current batch-major path: each lane reads the
same `dY`, the same `W_T`, and accumulates in the same K order. Only the staging
of the shared weight tile changes.

Production slot strides must come from `plan.slots[slot] / 4`, not from compact
tensor dimensions. The real batch-major buffers are laid out as:

```text
slotBuffer = [batch][slotFloats]
array<vec4f> stride = slotFloats / 4
```

The compact microbench can use `cin * P4` and `cout * P4` strides, but the
production emitter should use real plan slot strides exactly like
`pointwiseSharedWBatchForwardDispatch()`.

## First Implementation Shape

Keep the first fork plain `pw_bwd` only:

- support `step.kind === "pw_bwd"`;
- support `batch` in `[1, 3]`, with performance testing focused on `B=2` and
  `B=3`;
- preserve `step.accumulate`;
- do not handle `pw_bwd+gelu` or `pw_bwd+residual` in the first patch;
- fail loudly if `sharedWBackwardSteps` is combined with
  `fuseGeluBwdIntoPw` or `fuseResidualBwdIntoPw`.

`FUSE_PW_GELU=1` is forward-only and can remain enabled.

Later fused backward variants can reuse the same workgroup layout:

- `gelu_bwd -> pw_bwd`: stage `dY * geluGrad4(pre)` per lane, requiring a per-lane
  `preBase`.
- `residual_bwd -> pw_bwd`: add the residual-copy extra store per lane, requiring
  a per-lane `resDstBase`.

Do not include those until plain shared-W `pw_bwd` wins the full-chain gate.

## Target Shapes And Steps

The current `pointwise_report.ts` output at `BATCH=3` says the full pointwise
train family is:

```text
48 forward pointwise dispatches
48 backward pw_bwd dispatches
26.575 GFLOP pointwise train math
3445.13 MiB lower-bound staged global traffic
51648 pointwise workgroups
```

For this shared-W backward fork, target high-traffic repeated `pw_bwd` families
before low-spatial `8x8` shapes. Shared-W reduces repeated weight staging, but
it also reduces workgroup count by `B`; that is dangerous on the already-small
`8x8` layers that v18 split-K was designed to fix.

### Bring-Up / Microbench

Use a tiny allowlist that covers both output widths and accumulate behavior:

```text
30  256->768 @16x16  accumulate=false
32  768->256 @16x16  accumulate=false
13  1536->512 @8x8   accumulate=true
```

Run this for `B=2` and `B=3` in a compact z-batch vs shared-W bench before full
CLIP integration.

### First Full-Chain Allowlist

Start with all repeated `16x16` ConvFFN backward families:

```text
30,32,36,38,42,44,48,50,54,56,60,62,66,68,72,74,78,80,84,86
```

These are:

```text
256->768 @16x16, indexes 30,36,42,48,54,60,66,72,78,84
768->256 @16x16, indexes 32,38,44,50,56,62,68,74,80,86
```

At `B=3`, those two families are `6.040 GFLOP` and about `750.00 MiB` of
lower-bound staged traffic. The shared-W lower-bound weight-read saving is about
`240 MiB` across the family.

### Second Allowlist

If the first allowlist wins, add the repeated `32x32` families:

```text
95,97,101,103,107,109,113,115,119,121,125,127
```

These are:

```text
128->384 @32x32, indexes 95,101,107,113,119,125
384->128 @32x32, indexes 97,103,109,115,121,127
```

At `B=3`, this adds `3.624 GFLOP`, about `468.00 MiB` lower-bound staged
traffic, and about `144 MiB` of lower-bound weight-read saving.

### Risky Third Allowlist

Only after the above gates, test the late `8x8` family against split-K as an
alternative, not stacked with it:

```text
8,13,17,22,6,15
```

The `1536->512 @8x8` steps `8,13,17,22` have very high K and low workgroup
count. Shared-W saves weight loads, but it also changes `96` B=3 workgroups into
`32` larger workgroups. Split-K may be the better answer for those layers.

Avoid early `64x64` stem-adjacent steps in the first shared-W policy. They have
many workgroups already, lower channel widths, and more source/write traffic
relative to weight traffic.

## Source Changes For A Future Patch

### `src/clip/vision_batch_pointwise.ts`

Add a backward equivalent of the forward shared-W emitter:

```ts
export function pointwiseSharedWBatchBackwardDispatch(
  plan: TrainPlan,
  s: PwBwdStep,
  batch: number,
  weightPrecision: WeightPrecision = "f32"
): DispatchSpec
```

Implementation details:

- import `PwBwdStep` and `TrainPlan` as types from `vision_bwd_wgsl.ts`;
- use `weightsDecl(0, weightPrecision)`;
- bind `src` as grad slot `s.dY` and `dst` as grad slot `s.dX`;
- use `slotStride4(plan, s.dY, "dY")` and `slotStride4(plan, s.dX, "dX")`;
- return buffers `[{kind:"weights"}, {kind:"slot", slot:s.dY}, {kind:"slot", slot:s.dX}]`;
- label with a prefix that profiling can group separately, for example
  `pw_bwd_shared_w B3 256->768 @16x16`.

Optionally add compact helper dispatches for microbenching:

```ts
pointwiseZBatchBackwardDispatch(s, batch)
pointwiseSharedWBatchBackwardCompactDispatch(s, batch)
```

Those should use compact `[batch][tensor]` strides, mirroring the existing
forward compact bench.

### `src/clip/vision_batch_wgsl.ts`

Extend `BatchDispatchOptions`:

```ts
sharedWBackwardSteps?: ReadonlySet<number>;
```

Change `batchTrainDispatches()` so backward dispatch generation has a
batch-aware path before generic `batchSpec()`:

```text
if no sharedWBackwardSteps:
  current planBwdDispatches(plan, opts).map(batchSpec)
else:
  require !fuseGeluBwdIntoPw && !fuseResidualBwdIntoPw for v1
```

## Likely Risks

1. Workgroup memory is at the 16 KiB safe envelope for `B=3`. This may reduce
   occupancy enough to erase the saved weight reads.

2. Workgroup count drops by `B`. That is especially risky for `8x8` layers,
   where the current path already has few workgroups. Split-K may be better
   there.

3. `local_invocation_index` is a trap in a 3D workgroup. The shader must compute
   `li = lid.y * 8u + lid.x` so every batch lane stages its own full `xS` tile.

4. W loads may already hit cache. If global W traffic is not the bottleneck,
   shared-W can be flat or slower despite lower theoretical traffic.

5. `lane == 0` does all W staging. Other lanes stage only X and wait at the
   barrier. If this creates a long barrier tail, the variant can lose.

6. The forward shared-W path did not become a default win. Backward is worth
   testing because `pw_bwd` is hot, but this history argues for strict gates and
   shape allowlists.

7. Fused backward options can silently hide or replace selected `pw_bwd` steps.
   The first fork should reject `SHARED_W_BWD_STEPS` combined with
   `FUSE_GELU_BWD_PW=1` or `FUSE_RESIDUAL_BWD_PW=1`.

8. `BATCH=9` is not a target. The existing shared-W forward code caps at `B=3`
   because of workgroup memory; this fork should do the same unless a separate
   chunked-lane design is added.

9. Labeling matters. If `pw_bwd_shared_w` is grouped as generic `pw_bwd`, the
   timestamp result will be hard to interpret.

## Decision

Implement this as a narrow exact-math, batch-aware `pw_bwd` fork, not as a
mutation of the baseline backward emitter. Reuse one staged `wS` tile across
`B=2` or `B=3` lanes with `local_invocation_id.z`, keep one per-lane `xS` tile,
and gate it with explicit `SHARED_W_BWD_STEPS`.

The first serious allowlist should be the repeated `16x16` backward ConvFFN
families, not the `8x8` split-K candidates. If the `16x16` family fails the full
CLIP train matrix, keep the fork as research and do not broaden it.
